title: doorman agent
description: Continuous monitoring daemon with local logging or cloud alerts
---

## Overview

`doorman agent` runs a continuous monitoring loop. On each cycle it collects metrics and either logs them locally (local mode) or sends them to the Doorman backend for persistent monitoring and alerts (API mode).

```bash
# Local mode — log metrics as structured JSON, no external calls
doorman agent --local

# API mode — after running doorman login, no flags needed
doorman agent
```

Stop with `Ctrl+C` or `SIGTERM`. The agent handles both gracefully.

---

## Flags

| Flag | Default | Description |
|------|---------|-------------|
| `--config`, `-c` | — | Path to a YAML config file |
| `--interval` | `15` | Seconds between collection cycles |
| `--local`, `-l` | off | Log metrics locally, skip all API calls |
| `--token` | — | API key override (usually not needed after `doorman login`) |
| `--no-color` | off | Disable ANSI colors in log output |

---

## API key resolution

The agent finds your API key automatically in this order:

1. `--token` flag
2. `DOORMAN_API_KEY` environment variable
3. `~/.doorman/config` — saved by `doorman login` ← **recommended**
4. `api_key` in your `config.yaml`

After running `doorman login`, you never need to set an API key manually.

## Local mode

The best way to verify what the agent collects before connecting it to the backend.

```bash
doorman agent --local
```

Each cycle emits structured JSON to stdout:

```json
{
  "timestamp": "2026-06-15T14:22:01Z",
  "total_pending": 47,
  "total_active": 6,
  "saturation_pct": 37.5,
  "redis_connected": true,
  "celery_connected": true,
  "queues": [
    {"name": "celery", "depth": 12, "oldest_task_age_seconds": 3.2},
    {"name": "emails", "depth": 35, "oldest_task_age_seconds": 88.1}
  ],
  "workers": [
    {"name": "api-worker", "active_tasks": 3, "concurrency": 8, "is_alive": true}
  ]
}
```

Pipe to `jq` for filtering:

```bash
doorman agent --local | jq '.queues[] | select(.depth > 100)'
```

---

## API mode

API mode sends metrics to the Doorman backend on every cycle. The backend evaluates findings and sends Slack/email alerts when thresholds are breached.

```bash
# After doorman login — API key is loaded from ~/.doorman/config automatically
doorman agent

# Or with an explicit token
doorman agent --token sk_your_key_here
```

On startup the agent validates the API key. If invalid, it exits immediately with a clear error.

## Running as a background service

### systemd

```ini
# /etc/systemd/system/doorman-agent.service
[Unit]
Description=Doorman Agent — Celery queue monitor
After=network.target

[Service]
User=app
EnvironmentFile=/etc/doorman/env
ExecStart=/usr/local/bin/doorman agent --config /etc/doorman/config.yaml
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable --now doorman-agent
```

### Docker

```dockerfile
FROM python:3.12-slim
RUN pip install doorman-agent
CMD ["doorman", "agent", "--local"]
```

```bash
docker run -e REDIS_URL=redis://your-redis:6379/0 \
           -e CELERY_BROKER_URL=redis://your-redis:6379/0 \
           -e DOORMAN_API_KEY=sk_your_key \
           doorman-agent
```

### Docker Compose

```yaml
services:
  doorman:
    image: python:3.12-slim
    command: sh -c "pip install doorman-agent && doorman agent"
    environment:
      REDIS_URL: redis://redis:6379/0
      CELERY_BROKER_URL: redis://redis:6379/0
      DOORMAN_API_KEY: sk_your_key
    depends_on:
      - redis
    restart: unless-stopped
```
